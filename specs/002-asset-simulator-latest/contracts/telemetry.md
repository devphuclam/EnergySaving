# Telemetry Contracts

## Internal generated-measurement contract

Input contains:

- `measurement_id` - generated by Acquisition before calling Telemetry
- `source_id`
- `simulator_run_id`
- `point_id`
- `mapping_id`
- `mapping_version`
- `source_sequence`
- `algorithm_id`
- `algorithm_version`
- `simulator_configuration_id`
- `configuration_version`
- `source_timestamp`
- `numeric_value`
- `unit_code`
- `producer_identity`
- `correlation_id`
- `lineage_id`

Trusted internal producer identity is contextual, not an external transport credential. Telemetry
generates `received_at`, `processing_at` and quality/reason classification.

Telemetry MUST NOT generate a replacement Measurement ID. A missing, malformed or tuple-mismatched
`measurement_id`, or an untrusted producer, fails before identity reservation and is never falsely
reported as Duplicate.

## Stable terminal outcome registry

Telemetry extends `telemetry.measurement_identity` as the immutable terminal ingestion-result
registry. `measurement_raw` remains Accepted-only. The registry stores:

- `measurement_id` UUID primary key;
- immutable request identity: source, Run, Point, Mapping/version, source sequence,
  algorithm/version and Simulator configuration/version;
- `request_fingerprint`: SHA-256 of the canonical identity, value and provenance payload, excluding
  retry transport metadata;
- `final_classification`: Accepted or Rejected;
- `measurement_persisted` boolean and nullable `persisted_measurement_id`;
- nullable `quality_code`, `reason_code`, `rejection_code`, `latest_advanced`;
- `completed_at`;
- safe original correlation and lineage references.

Constraints:

- `source_sequence >= 0`; mapping/configuration/algorithm versions are positive.
- `(simulator_run_id, point_id, source_sequence)` is unique in addition to `measurement_id`.
- Accepted requires `measurement_persisted=true`,
  `persisted_measurement_id=measurement_id`, quality Good/Uncertain/Bad,
  `rejection_code IS NULL`, and non-null `latest_advanced`.
- Rejected requires `measurement_persisted=false`, no persisted Measurement reference, no quality,
  non-null `rejection_code`, and null `latest_advanced`.
- Registry rows are immutable. Registry retention is at least as long as the retry and Measurement
  retention horizon.

The fingerprint prevents a reused `measurement_id` with different identity/value/provenance from
being treated as a valid Duplicate; that case returns `IDEMPOTENCY_CONFLICT` without mutating the
original result.

## Sequential validation and atomic persistence

1. Confirm trusted internal producer context. Failure is non-addressable and creates no registry row.
2. Parse required identity fields and `measurement_id`; require UUIDv5 shape and recompute the exact
   UUID from the canonical Simulator tuple. Missing, malformed or mismatched identity creates no
   registry row.
3. Compute the request fingerprint and perform a non-locking immutable registry lookup. A matching
   row returns Duplicate plus the exact stored original result, even if owner state later changed.
   A different fingerprint returns `IDEMPOTENCY_CONFLICT`.
4. Validate remaining static payload schema, timestamp and lineage. From this point, a failure has a
   valid addressable identity and produces a stable Rejected result.
5. For a new candidate, open a host-coordinated REPEATABLE READ transaction and lock in order:
   Organization operational Point/ancestors -> Catalog Source/Mapping/Metric/Unit -> Telemetry
   identity/raw/Latest -> Integration outbox.
6. Recheck the registry after provider locks. A concurrent matching result returns Duplicate; a
   conflicting fingerprint returns `IDEMPOTENCY_CONFLICT`.
7. Owner, Catalog or payload failure after valid identity produces one Rejected terminal registry
   row atomically, without `measurement_raw` or Latest.
8. Accepted inserts the terminal registry result and `measurement_raw`, performs Latest
   compare-and-set, records `latest_advanced`, and writes outbox in the same commit.
9. A unique-race loser reloads the immutable winner and returns Duplicate or
   `IDEMPOTENCY_CONFLICT`. No in-progress registry state is exposed.

The step 3 preflight is immutable and non-locking, so it does not change the global lock order.

## Result

- **Accepted**: response disposition Accepted with stored `original_result`;
  raw Measurement is persisted as Good, Uncertain or Bad and includes reason/Latest-advanced state.
- **Rejected**: response disposition Rejected with stored `original_result`; no raw Measurement
  exists, but the identity-addressable terminal result is committed.
- **Duplicate**: response disposition Duplicate with `original_result` copied from the registry:
  `final_classification`, `measurement_persisted`, persisted Measurement reference when present,
  quality, reason, rejection code, `latest_advanced`, `completed_at`, and safe original
  correlation/lineage references. It creates no second row and no business counter increment.

A Rejected duplicate is returned from the stored terminal result, never reconstructed from a raw
Measurement. A well-formed trusted Simulator request with a valid `measurement_id` receives one
stable terminal Accepted or Rejected result.

P-001: Good and Uncertain are Latest-eligible; Bad is persisted but not eligible; No Data is never a
Measurement. P-002: future skew beyond configurable 300 seconds is preserved Uncertain with
`SOURCE_TIMESTAMP_FUTURE` and remains eligible. Out-of-range is Accepted Bad with
`VALUE_OUT_OF_RANGE`.

P-003: compare source timestamp, then sequence when supplied and resolving an equal timestamp, then
processing timestamp, then Measurement ID. Simulator always supplies sequence. Older/out-of-order
history remains stored without Latest regression. Duplicate identity is not out-of-order data.

## Acquisition finalization contract

Acquisition finalizes a Pending attempt in one transaction. It stores the response disposition and
stable `original_result`, transitions Pending -> Completed once, and increments exactly one
Accepted or Rejected run counter according to stored `final_classification`. Duplicate is not a
counter classification. Re-finalizing the same result is a no-op; a different result is an invariant
conflict requiring reconciliation.

## Query surface

- GET /api/v1/points/{pointId}/latest
- GET /api/v1/points/{pointId}/source-health
- GET /api/v1/sites/{siteId}/points/current

Responses contain Point identity, value, unit, source/received timestamps, quality/reason, elapsed
time, Run status, last received time, Generated/Accepted/Rejected counts and Source Status. Scope is
applied before lookup/paging. No accepted observation returns Latest absent plus NoData, never zero.

## Source Status

Physical projection is `telemetry.point_source_status`. Worker uses server time and last accepted
received time: Online when elapsed <= expected interval; Stale when above expected and <= no-data
threshold; NoData when above threshold. Decommissioned overrides Suspended, which overrides elapsed
status. Evaluation is idempotent and validates owner snapshot/version before update. A real status
transition emits an event; repeated evaluation does not.

## Events

`MeasurementAccepted.v1` contains identifiers, quality/reason, timestamps and lineage.
`PointLatestAdvanced.v1` contains old/new IDs and the ordering tuple.
`PointSourceHealthChanged.v1` is emitted only on a real status transition. All use
`integration.outbox_event` and `inbox_message` deduplication.
