Tóm tắt điều hànhKiến trúc được chọn là Modular Monolith có boundaries rõ, triển khai thành hai process trung tâm: Application API và Background Worker. Web SPA phục vụ người dùng; PostgreSQL là source of truth cho metadata, telemetry, rule state, alert, jobs, outbox/inbox và audit; file được lưu qua abstraction để MVP dùng filesystem nội bộ và có thể chuyển object storage sau này. Cách chọn này đáp ứng quy mô POC nhỏ, giảm số thành phần phải vận hành và tránh phân tán transaction khi nghiệp vụ còn đang thay đổi.1.1. Architecture summaryKhía cạnhQuyết địnhStyleModular Monolith + Clean/Hexagonal boundaries trong từng module; CQRS-lite cho command/query.RuntimeReact/TypeScript Web SPA; ASP.NET Core LTS Application API; .NET Worker process.DataPostgreSQL với schema ownership theo module; telemetry partition theo thời gian; latest và aggregate read models.AsyncTransactional outbox/inbox và database-backed job queue; chưa dùng external broker.SecurityLocal IAM trong MVP-1, server-side RBAC + site/area scope, cookie/session cho web, source credential cho ingestion.DeploymentOn-premise containerized reference deployment; App host và DB host tách ở production; không Kubernetes.OTEdge Collector riêng ở MVP-2; Modbus read-only, buffer >=7 ngày, outbound HTTPS only.ExtensibilityAdapter/port cho source, rule evaluator, notification channel, report generator, file storage và integration publisher.OperationsHealth, structured logs, metrics, correlation ID, visible job status, backup/restore và runbook.1.2. Vì sao không chọn microservices ngayQuy mô POC chỉ 8-20 điểm đo, khoảng 11.520-28.800 bản ghi/ngày; database và worker đơn giản đủ xử lý.Domain, workflow, responsibility và dữ liệu thật chưa baseline; microservices làm tăng contract, deployment, monitoring và failure mode trước khi boundary ổn định.Alert, rule state, audit và permission cần consistency cao; monolith giảm distributed transaction và operational burden.Đường tách vẫn được giữ bằng module ownership, event contracts, outbox/inbox, separate worker và architecture tests.Quyết định chủ đạoXây một hệ thống đơn giản để dùng được ngay, nhưng tổ chức code và data ownership như thể mỗi module có thể được tách sau này.
